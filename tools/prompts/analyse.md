## Two-Pass Analysis

### Pass A — Build the Usage Map

Before looking at any component in isolation, scan every file in the directory and record:

- Every component call site: which file calls it, on which line, and what classes are on the immediate JSX wrapper at that call site
- What layout context the parent establishes for its children at that site (flex direction, gap, padding, rounding, text style, width, overflow, bg…)

Store this map internally — you will reference it throughout Pass B.

### Pass B — Analyse Each File

For every `.tsx` / `.jsx` / `.js` file, identify every proposed change in these categories:

#### B1 — Intra-component redundancy

Classes on the **same element** that are:

- Duplicated (`flex flex-row flex-row`)
- Conflicting on the same axis (`mt-4 mt-6` — note both, flag if conditional)
- Redundant with Tailwind's own defaults (`block` on a `<div>`, `font-normal` with no weight override)
- Visually inert in context (`relative` with no `absolute` descendants in the subtree)

> Note: do NOT flag duplicates or conflicts already caught by an intra-file Tailwind linting plugin — focus on what a plugin cannot see.

#### B2 — Heritage redundancy (cross-component)

Classes on a **child's root element** that are already established by its parent context at the call site:

- Parent uses `flex flex-col gap-6` → child root has `mt-6` (spacing already handled)
- Parent has `rounded-xl overflow-hidden` → child root repeats `rounded-xl` (clipped anyway)
- Parent sets `text-sm text-gray-500` → child root re-declares same (inherited)
- Parent forces `w-full` on children → child root also has `w-full`
- Parent sets a background → child root repeats the same background token

For each instance, decide and state the **fix direction**:

- **Fix child** — remove from the child's root (child was over-declaring)
- **Fix parent** — remove the wrapper class at the call site (parent was over-wrapping)

If the component is called in multiple places with inconsistent parent contexts → mark as `⚠️ REVIEW` instead of proposing a removal.

#### B3 — Unnecessary wrapper divs

Divs that can be removed or unwrapped:

- No classes, no attributes, single child → unwrap
- Layout classes already handled by parent context → unwrap and move meaningful classes
- Spacing-only wrapper that can be expressed as margin/padding on the child
- Fragment-equivalent (no visual role)

For each: state what happens to any classes on the wrapper (moved to child, moved to parent, dropped because redundant).

#### B4 — Unknown / unverifiable

- Classes not recognisable as standard Tailwind or an obvious theme token → mark `⚠️ VERIFY`
- Classes inside `cn()`, `clsx()`, `classnames()`, or ternaries that cannot be fully traced → mark `⚠️ CONDITIONAL`
- Classes generated by string interpolation → mark `⚠️ DYNAMIC — do not touch`
- Components imported from outside the package boundary → mark `⚠️ EXTERNAL — black box, skip heritage analysis for this component`
