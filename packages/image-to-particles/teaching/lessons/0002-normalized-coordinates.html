Normalized Coordinates

Lesson 2 · 15 min

Every p5.js beginner writes circle(200, 200, 50) — hardcoded pixel values. That works fine on one canvas size. But change the canvas, and everything breaks: shapes go off-screen, proportions warp, your beautiful composition falls apart.

Normalization is the technique of designing in a unitless space (usually 0 to 1) and mapping to pixel coordinates at render time. It's the single most important pattern for writing resolution-independent sketches — and yes, it really is as "magic" as you've heard.

The problem: hardcoded pixels

// Works perfectly at 400×400...
circle(200, 200, 80);
circle(150, 150, 40);
circle(250, 150, 40);

// ...completely broken at 200×600 or 800×300

Here's the same design — two eyes and a mouth — at three different canvas sizes using hardcoded coordinates. Each sketch uses createCanvas(w, h) but the shapes don't adapt:

Notice: the face is perfectly centered at 400×400. At 200×200 everything is squeezed. At 600×200 the face is cut off and stretched. The logic is the same — but the output is wrong because pixel values are fixed.

The solution: think in proportions

Instead of 200 pixels from the left, think halfway across the canvas. p5.js gives you width and height for free inside your sketch: 

// Normalized — works at ANY size
let cx = width  * 0.5;        // center x
let cy = height * 0.5;        // center y
let r  = width  * 0.2;        // radius = 20% of width

circle(cx, cy, r * 2);         // face
circle(cx - r * 0.5, cy - r * 0.4, r * 0.5);  // left eye
circle(cx + r * 0.5, cy - r * 0.4, r * 0.5);  // right eye

Same code, any canvas. The normalized version produces a proportional face no matter the aspect ratio. This is why normalization is the first tool professionals reach for.

Three strategies for normalization

1. Manual mapping (explicit)

Multiply each normalized coordinate by width or height. Most readable, no p5.js magic needed: 

function setup() {
  createCanvas(400, 400);
}

function drawFace(nx, ny, size) {
  // nx, ny, size are all 0–1
  let cx = nx * width;
  let cy = ny * height;
  let r  = size * min(width, height) * 0.5;

  circle(cx, cy, r * 2);
  circle(cx - r * 0.4, cy - r * 0.35, r * 0.4);
  circle(cx + r * 0.4, cy - r * 0.35, r * 0.4);
}

2. scale() transform (concise)

Set up a virtual coordinate system once. Everything inside is in your virtual units:

function setup() {
  createCanvas(400, 400);
  noLoop();          // static sketch
}

function draw() {
  // Map width → 100 virtual units, preserving aspect ratio
  let s = width / 100;
  scale(s);

  // Now coordinates are in 0–100 virtual space
  circle(50, 50, 30);   // face at center, diameter 30
  circle(40, 40, 10);   // left eye
  circle(60, 40, 10);   // right eye
}

3. Helper functions (cleanest for complex sketches)

// Reusable: add these at the top of any sketch
function nx(v) { return v * width; }
function ny(v) { return v * height; }
function nd(v) { return v * min(width, height); }

// Usage:
circle(nx(0.5), ny(0.5), nd(0.3));

The key insight: nd() vs nx() / ny()

A detail that trips people up: which dimension to use for size?

  • Position: use nx() for x (based on width), ny() for y (based on height).
  • Size: use nd() — based on min(width, height) — so circles stay circular.

If you use nx() for both width and height of a shape on a non-square canvas, you'll stretch it horizontally. nd() uses the smaller dimension, keeping proportions consistent while still scaling with the canvas.

How this relates to the three strategies from last session

Recall the three responsive-canvas strategies:

  1. Fixed resolution + CSS scale — render at 800×600, let CSS stretch/shrink it.
  2. Full responsive via windowResized — call resizeCanvas() on window resize, rewrite positions.
  3. Virtual coordinate system via scale() — use normalized coords mapped through p5 transforms.

Normalization is the engine that makes strategies 2 and 3 practical. Without normalization, strategy 2 forces you to manually reposition every shape on every resize. With normalization, windowResized just updates width/height, and your proportional calculations automatically produce correct positions.

The magic: normalization separates what to draw from how big the canvas is. Your design logic stays clean and mathematical. The canvas becomes just a parameter. That's it — that's the magic.

Practice

Open the sketch above in your own editor. Try these:

  1. Add a triangle to the normalized side — position it with nx(), ny().
  2. Create a grid of circles: 5 columns × 4 rows, filling the canvas proportionally.
  3. Write a nline(x1, y1, x2, y2) helper that takes all normalized coordinates and draws a line.
Remember: normalization isn't just for responsive layouts. Any sketch with parametric or mathematical positioning benefits from it — particle systems, noise fields, geometric patterns. It becomes second nature very quickly.
