Before — two leaky import paths

After — one interface, one seam

Problem

image-manipulator reaches into image-pipeline through two export paths (/manipulations and /PipelineGateway), and only uses pixel-level manipulations. Two thirds of the pipeline's capabilities — neighborhood and global — are unreachable through this consumer. If ManipulationDefinition changes, both packages break.

Solution

Consolidate to a single Pipeline module that exposes a narrow interface: run(definition[], source) → result. The gateway, registry, and scheduler become implementation details behind the seam.

Wins

  • Leverage: one interface for all 3 access types
  • Locality: pipeline internals change without touching consumers
  • Tests: the Pipeline interface is the test surface; swap gateway for a mock adapter
  • Deletion test: complexity concentrates in the pipeline module, not across two packages
  • One adapter today (image-manipulator) → seam justified by second consumer emerging

Interface stays small; implementation absorbs variant logic for all 9 components.

Problem

Nine UI components each define their own variant system (buttonVariants, badgeVariants, etc.), with different sets, naming, and defaults. This duplication causes 8 documented bugs (invisible error text, overlapping icons, always-true onChange). The interface for each component is nearly as complex as its implementation — shallow.

Solution

Extract a single variantConfig module. Each component declares its variant schema as data. One module applies them, resolves conflicts, and provides a single function: resolveVariants(config, props) → classNames.

Wins

  • Locality: all variant logic in one place — fix a bug once
  • Leverage: 9 call sites, one resolution interface
  • Tests: one seam to test instead of 9; 8 known bugs become 8 test cases
  • Deletion test: removing the old 9 variant files concentrates complexity in one module
  • Adding a new component's variants requires only a schema declaration

Absorb the generateSequence wrapper; deepen the rule seam

Worth exploring

in-process

packages/sequence-renderer/src/core/generateSequence.ts

packages/sequence-renderer/src/core/sequence-rules.ts

Before — shallow wrapper + noisy rule

After — rule logic deepened into rule module

Problem

generateSequence.ts is a 28-line pure function extracted for testability, but has zero tests — and the real bugs (collatzRule never terminating, fibonacciRule off-by-one) are in how the rules interact with the generation loop, not in isolation. The module is shallow: the interface is nearly as complex as the body.

Solution

Merge generateSequence into the rule module. The SequenceRule interface stays as the seam; the generation loop becomes an implementation detail. Rules gain the loop context so bugs like termination conditions are testable in one place.

Wins

  • Locality: sequence generation + rules in one module — bugs visible in one file
  • Leverage: callers import one module instead of two
  • Tests: the rule module's interface is the test surface; collatz bug gets a test case
  • Deletion test: deleting generateSequence.ts moves 28 lines into the rule module — complexity concentrates

Problem

Solution

Move the component and its helpers into apps/playground/src/features/image-to-particles/. Delete the package. If a second consumer emerges, extract then — the seam will be real, not hypothetical.

Wins

  • Leverage: zero interfaces to maintain for a single-consumer feature
  • Removes a package.json, build config, and versioning overhead
  • Deletion test: complexity vanishes (nothing depends on it)

Problem

Three packages (automa, graph-viz, image-pipeline) use Web Workers in three different patterns: Transferable buffers, string-template Blob URLs, and a worker pool. None share a common interface. Adding a new worker-based feature means picking a pattern from scratch.

Solution

Extract a @repo/worker-pool module with a WorkerPool interface. image-pipeline's PipelineGateway is a strong starting pattern — generalize from it. Each consumer provides an adapter (a concrete WorkerPool implementation).

Wins

  • Leverage: one worker interface, three consumers
  • Seam: swap worker implementations — main thread, worker pool, mock — behind the same interface
  • Tests: mock adapter for all worker-dependent code
  • Locality: worker lifecycle bugs fixed in one module

Top recommendation

Candidate 1: Deepen the image-manipulator→image-pipeline seam

Strong

image-pipeline is the deepest module in the monorepo — well-layered, with typed seams, a plugin registry, fusion scheduling, and worker pool management. But its consumer (image-manipulator) reaches past the intended interface through two separate import paths and ignores 2/3 of the pipeline's capabilities. Tackle this first because the foundation is already strong: you're fixing a leaky seam, not building new architecture. The payoff is immediate — a clean Pipeline interface means all three access types (pixel, neighborhood, global) become testable through one seam, and the image-pipeline refactoring plan (documented in Community 37 of the graph report) shows this was already a known concern. Second choice: the UI variant consolidation, which fixes 8 bugs and reduces duplication across 9 components.
